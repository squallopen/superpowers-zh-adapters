# v0.2.2 更严格工程习惯正式版

这是 `v0.2.2` 的正式版。

如果用一句话概括它和 `v0.2.1` 的区别，就是：
`v0.2.1` 更偏“把中文触发、中文文档输出和安装更新体验整理顺”，
`v0.2.2` 则进一步把“先看资料、先写计划、再实现、补测试、补文档、最后按证据交付”这条工程链条收紧了。

## 相比 v0.2.1，v0.2.2 主要加强了什么

### 1. 规划和设计阶段更清楚

- 更明确区分了 `brainstorming` 和 `writing-plans`
  `brainstorming` 更偏“先把需求、边界、方案想清楚”，
  `writing-plans` 更偏“直接产出实施计划、接口设计、数据结构、表结构、Redis / S3 设计、字段说明、OpenAPI 骨架”等文档。

- 加强了中文触发词
  现在像 `接口设计`、`API设计`、`数据结构设计`、`数据模型设计`、`表结构设计`、`Redis设计`、`S3设计`、`接口文档`、`字段说明` 这类说法，更容易稳定命中 `writing-plans`。

- 更适合按 `TODO.md` 驱动规划
  如果你把 `TODO.md`、backlog 文件或工作清单作为本次需求来源，
  现在会更倾向于按原顺序处理，从第一个可执行事项开始；
  如果第一项太小，再只向后合并紧邻、同主题的事项，
  而不是先反问你“要规划哪个 feature”。

### 2. 实现前的信息检索更严格

- 在开始写代码前，更强调先检索、再阅读
  优先使用工具自带的文本搜索能力，或者它自己判断最合适的检索方式，
  先定位相关内容，再只读命中的 `README` 章节、实施计划、设计说明、接口设计、数据结构 / schema、Redis / S3 设计、技术栈说明、运行说明等相关段落，
  不再鼓励整篇读取无关文档。

- 如果计划依赖的文档不存在、过期或互相矛盾，会更倾向于先停下来指出问题，而不是直接硬做。

### 3. 测试要求更完整

- 写完代码后，默认要求补单元测试
- 如果改动跨服务、跨接口、跨存储、跨流程，且仓库里具备条件，也要求补或运行集成测试
- 如果确实不适合做集成测试，需要明确写原因，而不是跳过不提

这部分不只体现在 `executing-plans`，也补到了 `subagent-driven-development` 和 `verification-before-completion` 相关约束里。

### 4. 文档回填和交付总结更严格

- 如果实现改动影响了接口、字段、行为说明、架构说明、运行说明或示例，要求在交付前把相关文档一起回填
- 最终总结时，要明确列出：
  新增了哪些文件、
  修改了哪些文件、
  读了哪些文档、
  回填了哪些文档、
  跑了哪些单元测试、
  跑了哪些集成测试、
  结果怎么样

- 如果相关测试没跑，要明确标 `NOT RUN`
- 如果测试失败，要明确标 `FAILED`
- 如果相关文档还没补，要明确标 `NOT BACKFILLED` 或 `BACKFILL REQUIRED`

换句话说，`v0.2.2` 更强调“按证据交付”，不鼓励只凭叙述说“已经测过了”“文档已经补了”。

### 5. 计划文件的说法更不容易把人带偏

- 现在更明确：
  `writing-plans` 先产出一份计划或设计文档，
  `executing-plans` 再明确按那一份文档继续执行

- 不再把计划文件绑定到某个固定目录
  如果你指定了保存位置，就按你的要求来；
  如果你没指定，就优先沿用仓库现有的文档习惯。

这更符合真实项目里的使用方式，也更适合你们这种 `TODO.md -> 计划 -> 执行` 的工作流。

### 6. 上游能力也同步到了更新版本

- vendored 的 `obra/superpowers` 已刷新到 `v5.0.5`
- 这让中文适配层继续跟着上游演进，不是停留在较早的 `v5.0.2`

## 谁更适合直接升到 v0.2.2

更适合下面这些场景：

- 你希望 AI 先看资料、再动手，而不是直接猜着改
- 你希望 AI 写完代码后主动补测试、补文档、补结果总结
- 你希望它按 `TODO.md` 的顺序来，不要总是重选范围
- 你希望最终交付更容易核对，而不是靠一句“应该没问题”

## 如果你已经在用 v0.2.1

升级到 `v0.2.2` 后，你最直接能感受到的变化通常是：

- 更少直接跳过规划
- 更少“没跑也说跑了”的模糊表述
- 更少“代码改了但文档没跟”的情况
- 更少在 `TODO.md` 工作流里反复问你“到底要做哪个 feature”
- 更清楚地区分“已经完成”“没跑”“失败”“还没回填”

## 说明

- `v0.2.2` 吸收了此前 `v0.2.2-beta.1`、`v0.2.2-beta.2`、`v0.2.2-beta.3` 的主要约束和说明整理结果。
- 如果你之前已经在试 beta，这次可以直接切到正式版。
