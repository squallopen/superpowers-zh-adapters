# v0.2.2 TODO 驱动工作流正式版

这版最重要的变化，不是再堆一层术语，而是把原来比较繁琐的行为模式改成更直接的 `TODO -> 计划 -> 执行`。

## 相比 v0.2.1，重点改了什么

- `writing-plans` 现在可以直接把 `TODO.md`、backlog 文件、worklist 当需求来源
- 默认按清单原顺序处理，从第一个可执行事项开始
- 如果第一项太小，只向后合并紧邻、同主题的事项，不再乱跳
- 不再默认反问你“要我规划哪个 feature”
- `executing-plans` 更明确是“按已经写好的那份计划执行”，不是回头再从 TODO 里重选范围
- 计划文件不再默认绑死到固定目录；你指定路径就按你的路径，没指定就沿用仓库原来的文档习惯

## 直接说，就是把什么改掉了

原来更容易出现这种体验：

- 你明明已经给了 `TODO.md`
- 它还是继续追问“这次到底规划哪个功能”
- 或者又自己重新挑范围
- 甚至计划写完后，执行阶段又重新选一遍

`v0.2.2` 重点就是把这类繁琐、重复、容易跑偏的行为压下去。

## 现在更推荐的用法

先让它按 TODO 写计划：

```text
/superpowers-writing-plans
把 doc/codex/TODO.md 视为本次需求来源。
按 TODO 顺序处理，从第一个可执行事项开始；如果这一条太小，就只向后合并紧邻、同一主题的事项。
不要重新反问我要规划哪个 feature。
```

再让它按刚写好的计划执行：

```text
/superpowers-executing-plans
按刚写好的那份计划执行，不要重新从 TODO 里改选范围。
```

## 另外一起带上的增强

- 实现前更强调先检索，再只读命中的 README / 设计 / 接口 / 数据结构 / Redis / S3 段落
- 写完代码后更强调补单元测试，能做时补集成测试
- 交付时更明确区分 `NOT RUN`、`FAILED`、`NOT BACKFILLED`、`BACKFILL REQUIRED`
- 中文触发词也补强了，像 `接口设计`、`数据结构设计`、`Redis设计`、`S3设计`、`字段说明` 这类更容易直接命中
- vendored 上游 `obra/superpowers` 已同步到 `v5.0.5`

## 适合谁

- 你本来就是按 `TODO.md` 推进工作
- 你不想再被反复追问“到底要做哪个 feature”
- 你希望计划和执行能真正接起来，不要中间再重选范围
