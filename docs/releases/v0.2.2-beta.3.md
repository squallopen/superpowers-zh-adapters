# v0.2.2-beta.3 面向更严格工程习惯的测试版

这是一个继续面向更严格工程习惯用户的测试版。

`v0.2.1` 仍然是当前稳定版；如果你希望 AI 在实现前先做相关资料检索、实现后补测试和文档，并在交付时给出更完整的结果总结，可以试用这个版本。

这版延续 `v0.2.2-beta.2` 的方向，重点补的是“计划怎么接着往下执行”这件事，让 TODO 驱动的工作方式更顺手，也更少跑偏。

## 这版主要补了什么

- 刷新了上游 `obra/superpowers` 到 `v5.0.5`
  这一版把仓库里 vendored 的上游内容同步到了更新版本，方便后续继续跟上游对齐适配。

- 加强了 `writing-plans` 对 TODO / backlog / worklist 的理解
  当你明确把 `TODO.md`、backlog 文件或任务清单当作本次需求来源时，
  现在更倾向于：
  按原顺序处理、
  从第一个可执行事项开始、
  如果第一项太小，再只向后合并紧邻、同主题的事项，
  而不是先反问“你要我规划哪个 feature”。

- 把“计划文件路径”讲清楚了
  现在文档和适配层都更明确：
  不要求把计划固定放在某个目录里；
  如果你指定了保存位置，就按你的要求来；
  如果你没指定，就优先沿用仓库现有的文档习惯。

- 把 `executing-plans` 的使用边界讲得更清楚了
  它更适合“按已经写好的那份计划继续执行”，
  而不是再次从 `TODO.md` 里重选范围。
  也就是说，先写计划，再明确按哪一份计划执行，会更稳。

## 适合谁

这个测试版更适合下面这类用户：

- 已经习惯先列 `TODO.md`，再让 AI 先写计划、后执行
- 不希望 AI 反复追问“到底要做哪个 feature”
- 不希望计划文件被默认绑死在某个固定目录
- 希望中文工作流更贴近日常项目协作习惯

## 需要注意

- 这是 `beta`，意味着它比 `v0.2.1` 更严格，也更强调“先计划、再执行、再验证”。
- 如果你更喜欢轻量、直接、少约束的行为，建议继续使用 `v0.2.1`。
- 如果你愿意帮助验证这套更严格的规划和执行约束是否更顺手，欢迎试用这个版本并反馈实际体验。
